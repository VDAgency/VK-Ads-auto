# DEPLOY.md — правила деплоя (ОБЯЗАТЕЛЬНО к прочтению перед любым деплоем)

> **Жёсткое правило для агента.** В начале КАЖДОЙ сессии (включая параллельные) перед любым действием, связанным с деплоем, SSH или ключами — перечитай этот файл. **Никогда не выдумывай имена ключей, пути и шаги деплоя по памяти.** Источник правды — только этот файл. Если данных не хватает — спроси, не угадывай.

---

## 0. Принцип

- Разработка — только на **локальной машине**. Сервер ничего «своего» не накапливает.
- На сервер код попадает **только через git**: правим локально → commit → push → проверки на GitHub → деплой.
- **Деплой происходит только если все проверки зелёные.** Красное — не деплоим.
- Репозиторий **публичный** → секреты в git не попадают НИКОГДА (см. раздел 4).

---

## 1. Два SSH-ключа (имена фиксированы — заполнить один раз, дальше не менять)

> ВПИШИ реальные имена ключей ниже. Агент обязан использовать РОВНО эти имена и никогда не генерировать/переименовывать их сам.

### Ключ 1 — Локальная машина ↔ Сервер (доступ разработчика)
- **Имя ключа:** `<VK-Ads-auto>`
- Приватная часть: на локальной машине (`~/.ssh/<имя>`).
- Публичная часть: на сервере в `~/.ssh/authorized_keys`.
- Назначение: вручную зайти на сервер по SSH — внести/обновить секреты в серверном `.env`, поднять n8n, посмотреть логи, отладка.
- Команда подключения: `ssh -i ~/.ssh/<имя> <user>@<SERVER_IP>`

### Ключ 2 — GitHub Actions ↔ Сервер (автоматический деплой)
- **Имя ключа:** `<VK-Ads-auto-deploy>`
- Приватная часть: в **GitHub → Settings → Secrets → Actions** под именем `SSH_DEPLOY_KEY` (НЕ в коде).
- Публичная часть: на сервере в `~/.ssh/authorized_keys`.
- Назначение: GitHub Actions заходит по SSH на сервер и выкатывает изменения.
- Репозиторий публичный → сервер тянет код по HTTPS, отдельный deploy-key для `git pull` не нужен.

**Реквизиты сервера (заполнить):** `SERVER_IP = <...>`, `SSH_USER = <...>`, путь проекта на сервере `<...>`.

## 2. Порядок деплоя (канонический, не отклоняться)

1. Правки сделаны **локально**.
2. Прогнать гейт качества локально: `ruff check .` + `ruff format --check .` + `mypy .` + `pytest -q` — всё зелёное.
3. `git add -A && git commit -m "..."` → `git push` в ветку `main`.
4. На GitHub автоматически:
   - CI прогоняет тот же гейт (ruff/mypy/pytest);
   - **Aikido** проверяет код через GitHub-интеграцию (НЕ локально по ссылке — локальный режим ломается из-за VPN).
5. **Только если всё зелёное** запускается job деплоя:
   - GitHub Actions заходит на сервер по SSH (Ключ 2, `SSH_DEPLOY_KEY`);
   - на сервере: `git pull` (HTTPS, публичный репо) → `docker compose pull && docker compose up -d --build`;
   - применить миграции БД (если есть);
   - проверить здоровье: эндпоинт `/health` отвечает 200.
6. Верификация: проверить `/health` и логи (`docker compose logs --tail=100`).

> Если проверки красные — деплой НЕ запускается. Чинить локально, повторить с шага 1.

## 3. n8n при деплое

n8n — отдельная ветка, разворачивается НЕ так, как наш код. Полные правила — в `docs/N8N.md`. Здесь только связка: воркфлоу n8n едут как JSON через git и накатываются на сервере отдельным шагом (см. N8N.md), а не через `docker compose up` нашего приложения.

---

## 4. Секреты (публичный репозиторий — критично)

- В git НЕ попадает ничего секретного. `.gitignore` закрывает `.env`, ключи, креды.
- **Секреты нашего кода** (DB-пароль, токен бота, ключи VK/Google/Senler, `N8N_ENCRYPTION_KEY`) → в серверном `.env`. Их вносит агент/разработчик по SSH (Ключ 1) из локального файла секретов, либо GitHub Actions пишет их из Actions Secrets. **В репозиторий — никогда.**
- **Креды n8n** → вводятся руками в UI n8n (см. `docs/N8N.md`), в `.env` и git не кладутся.
- `.gitattributes`: нормализация переносов строк (машина на Windows) — `* text=auto eol=lf`.

## 5. Что агент НЕ делает

- ❌ Не выдумывает имена SSH-ключей, IP, пути — берёт только из этого файла; нет данных → спрашивает.
- ❌ Не коммитит секреты, не печатает их в логи/вывод.
- ❌ Не деплоит при красных проверках.
- ❌ Не правит код «прямо на сервере» в обход git (сервер = производное от репозитория).
- ❌ Не сканирует Aikido локально «по ссылке» — только через GitHub-интеграцию.
