> Назначение: дать разработчику весь контекст «почему так» — какие решения приняты, почему, и что было до них (помечено «УСТАРЕЛО», чтобы не путаться в отменённых развилках). Это компаньон-справка к `PROJECT.md` / `BRIEF_SPEC.md` / `ROADMAP.md`, а не источник задач.

Формат записи: **Решение → Контекст → Почему → Статус.**

## 1. Коммерция и формат сделки

### 1.1. Цена 50 000 ₽, единая за оба блока ✅
- **Контекст:** изначально КП было на 85 000 ₽; на созвоне клиент попросил снизить.
- **Почему:** осознанный глубокий дисконт ради портфолио-кейса; цена едина за Блок 1 + Блок 2 без разбивки по этапам в смете.
- **Статус:** ✅ актуально.

### 1.2. Оплата 50/50 ✅
- **Решение:** 25 000 ₽ аванс при подписании + 25 000 ₽ после сдачи Блока 1 (MVP).
- **Почему:** второй платёж привязан к сдаче MVP, чтобы вся сумма была получена к готовности бота; Блок 2 идёт в рамках той же суммы как бонус клиенту.
- **Статус:** ✅ актуально. Аванс получен, проект стартовал.

### 1.3. ⛔ УСТАРЕЛО: 85 000 ₽ и схема 40/30/30
- Раньше: 85 000 ₽, оплата 40/30/30 (34 000 / 25 500 / 25 500), три варианта/этапа.
- **Статус:** ⛔ заменено пунктами 1.1–1.2.

### 1.4. Блок 2 — подарок в рамках цены ✅
- **Решение:** веб-интерфейс + клиентский портал + рефералка выполняются без доп. оплаты.
- **Почему:** осознанный жест; при росте числа клиентов ручной учёт рефералок был бы для клиента головной болью.
- **Статус:** ✅ актуально.

## 2. Интеллектуальная собственность

### 2.1. Взаимная неисключительность ✅
- **Решение:** Заказчик получает широкую неисключительную лицензию — пользоваться, дорабатывать код, **продавать доступ как SaaS** своим ученикам/клиентам. Исполнитель сохраняет исключительные права и право переиспользовать архитектуру/код в других проектах и сферах.
- **Контекст:** Михаил просил исключительное право на архитектуру за собой.
- **Почему:** при текущей (заниженной) цене заложена неисключительность; исключительность — отдельный, кратно более дорогой продукт, т.к. лишает Исполнителя основной модели работы. Разъяснено клиенту, линия удержана.
- **Статус:** ✅ актуально (раздел 6 договора).

## 3. Архитектура и интеграция

### 3.1. Адаптер с переключателем kotbot / прямой VK ✅
- **Контекст:** у kotbot **нет API** (подтверждено их ответом).
- **Решение:** слой-адаптер `PlatformAdapter` с двумя реализациями — kotbot (Playwright-автоматизация интерфейса) и прямой VK Ads API; переключатель (дефолт + ручной) + health-check + фолбэк с обратной связью оператору.
- **Почему:** kotbot ценен только для распределения одного бюджета по нескольким площадкам; для VK-only это лишняя прослойка, а прямой VK стабильнее. Выбор пути финализируется после доступа к кабинету.
- **Статус:** ✅ актуально.

### 3.2. Маркировка — на стороне VK ✅
- **Решение:** маркировку (erid, передача в ЕРИР) делать не нужно — VK сам ОРД и маркирует рекламу автоматически при запуске через VK Рекламу.
- **Почему:** уход от kotbot к прямому VK не теряет авто-маркировку; от рекламодателя нужны только данные рекламодателя.
- **Статус:** ✅ актуально (вне скоупа разработки).

### 3.3. ⛔ УСТАРЕЛО: мультиплатформенность
- Обсуждалась идея единого баланса и распределения по VK/Avito/OK/WB/Ozon/YouTube/TikTok. Проверка показала: **ни одна площадка не даёт API авто-пополнения баланса**. Клиент решил отказаться от мультиплатформы и снизить цену.
- **Статус:** ⛔ вне скоупа сделки с Анастасией; возможна будущая доработка/трек Исполнителя (адаптер уже это позволяет архитектурно).

### 3.4. API-first / headless-ядро (входящая абстракция) ✅
- **Контекст:** в первой версии PROJECT.md был зафиксирован только исходящий адаптер (VK/kotbot); входящая сторона (как фронты обращаются к ядру) не была описана — бот мог «влезать в код» напрямую.
- **Решение:** вся бизнес-логика — только в ядре; ядро отдаёт версионированный внутренний API; бот, веб, mini-app, будущий мобайл — тонкие клиенты. Инвариант: логика в ядре, интерфейсы сменные.
- **Почему:** даёт несколько интерфейсов для одной Анастасии (бот + веб + mini-app) и задел под мобайл без переписывания ядра; стоит почти ничего, т.к. веб Блока 2 и так строится поверх API.
- **Статус:** ✅ актуально (PROJECT.md, разделы 1, 5).

### 3.5. Готовность к мульти-тенантности (швы сейчас, продукт — позже) ✅
- **Решение:** строим однотенантно (Анастасия = тенант №1), но закладываем швы: `account_id` на всех таблицах, изоляция по тенанту, конфиг интеграций per-account.
- **Почему:** ретрофитить мульти-тенант позже дорого и рискованно (утечки между клиентами); швы — дешёвая страховка. Сам продукт сейчас не строим.
- **Статус:** ✅ актуально (PROJECT.md, разделы 1, 5, 6).

### 3.6. Мультиплатформа — через исходящий адаптер, будущий трек ✅
- **Решение:** добавление площадок (Avito/OK и пр.) — новая реализация `PlatformAdapter`, ядро не трогаем. Не в текущем объёме.
- **Напоминание:** авто-пополнение баланса по API недоступно нигде — везде ручное.
- **Статус:** ✅ как задел; реализация — будущий трек.

## 4. Скоуп и поток

### 4.1. Оператор — только Анастасия ✅
- **Решение:** единственный пользователь системы — Анастасия; роли менеджеров/руководителя убраны.
- **Контекст:** изначально обсуждались роли (менеджеры + руководитель), затем упрощено.
- **Статус:** ✅ актуально.

### 4.2. ⛔ УСТАРЕЛО: роли «менеджеры + руководитель»
- Заменено пунктом 4.1.

### 4.3. Иерархия данных ✅
- **Решение:** Оператор → Клиенты (вкл. саму Анастасию) → Кабинеты (1 клиент : N) → объект рекламы. Клиентом может быть субагентство со своими клиентами.
- **Статус:** ✅ актуально.

### 4.4. Четыре цели ✅
- **Решение:** подписчики, сообщения, лид-форма, заявка через Senler. Прописаны как 4 цели (подарок клиенту).
- **Маппинг на VK:** подписчики и сообщения — объект «Сообщество» (цели «Подписаться» / «Написать сообщение»); лид-форма — отдельный объект; Senler — через «Написать сообщение» (отдельной цели «Senler» в VK нет).
- **Контекст:** ранее в КП фигурировал набор из 3 целей; добавлена 4-я и уточнён маппинг.
- **Статус:** ✅ актуально.

### 4.5. ⛔ УСТАРЕЛО: «MVP только на цель подписчики»
- Ранний черновик ограничивал MVP одной целью. Заменено: Блок 1 поддерживает все 4 цели.

### 4.6. Поток брифа ✅
- **Решение:** оператор вводит контакт клиента (email/телефон/tg — автоопределение) → система шлёт ссылку на бриф → клиент заполняет → система по расписанию читает форму (Google Forms → Sheets API, n8n) → уведомление оператору → раскладка → интерактивный приём креатива → сводка с «Подтвердить/Исправить» → запуск.
- **Статус:** ✅ актуально.

### 4.7. Правки `номер.значение` ✅
- **Решение:** при «Исправить» оператор шлёт правки построчно в формате `номер.значение`, несколько за одно сообщение.
- **Статус:** ✅ актуально.

### 4.8. Контроль бюджета — на стороне Анастасии ✅
- **Решение:** система не делает авто-пополнение/мониторинг бюджета; это задача Анастасии.
- **Почему:** автопополнения по API нет; мониторинг/напоминания — будущая платная доработка.
- **Статус:** ✅ актуально.

### 4.9. Креатив — интерактивный приём ✅
- **Решение:** поддержка фото / карусели / видео + текст (+ хэштеги где допустимо); приём пошаговый, форматы и лимиты берём из актуальной документации VK.
- **Почему:** креатив — самый сложный узел; требования VK меняются несколько раз в год.
- **Статус:** ✅ актуально (детали в `BRIEF_SPEC.md`).

Регистрация агентства — зона клиента; подготовим письмо с условиями. Создание кабинетов через API делаем именно ради этой автоматизации. **Пометить и уточнить у Анастасии.**
- **Готовая форма брифа от Анастасии** — запросить.
- **Размер реферальной скидки** — согласовать с Анастасией.
- **Точные поля VK Ads API** — сверять с живым Swagger при разработке.

## 7. Q&A процесса разработки (согласование PROJECT.md)

10. **Креатив?** → Интерактивный приём, форматы из документации VK.
11. **Агентский статус?** → Сейчас неизвестен; предусмотреть и уточнить у Анастасии.

## 8. Будущий трек Исполнителя — SaaS-продукт (вне договора с Анастасией) ✅

- **Решение:** ядро проектируется так, чтобы Исполнитель мог позже запустить на его основе собственный SaaS-продукт для сторонних рекламщиков (таргетологов, фрилансеров) — аналог клиентской базы kotbot, но удобнее.
- **Модель:** Анастасия пользуется своим инстансом на своём сервере по договору; Исполнитель разворачивает **отдельную копию кода** (отдельная папка, отдельный деплой, своё название) на своём сервере и развивает продукт. Переиспользование кода — право по п. 6.4 договора; данные Анастасии не затрагиваются (п. 6.5).
- **Тайминг:** не сейчас. Запуск через ~1–2 месяца после полной сдачи Анастасии — сначала вносятся гарантийные правки по её проекту (чтобы не править дважды), плюс у Исполнителя параллельные приоритеты (WeCom/Денис, Нейроотдел/Стас).
- **Сейчас под это сделано только архитектурно:** headless-ядро (3.4), швы мульти-тенанта (3.5), точка расширения площадок (3.6). Сама продуктовая обвязка (регистрация/биллинг тенантов, онбординг, маршрутизация, админка, харденинг изоляции) — отдельный трек в `ROADMAP.md` → «Будущие треки».
- **Статус:** ✅ зафиксировано как задел; реализация отложена.
