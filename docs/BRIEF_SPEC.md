# BRIEF_SPEC.md — Спецификация разработки брифа

> Назначение документа: дать разработчику **метод** сборки брифа, а не готовый список вопросов «с потолка». Бриф — это производная от реальных параметров рекламного кабинета VK. Разработчик идёт по процедуре ниже, сверяется с актуальной документацией и живым кабинетом, и только потом формирует вопросы. Это закрывает риск «придумали вопросы, а под создание кабинета их не хватает».

Связанные документы: `PROJECT.md` (общая картина), `ROADMAP.md` (где этот этап в плане работ), `DECISIONS.md` (почему так решили).

---

## 0. Главный принцип

**Каждый вопрос брифа должен быть обоснован параметром, который реально нужен VK для (а) создания кабинета, (б) настройки кампании, (в) настройки группы (таргетинг), (г) создания объявления (креатив).**

Если параметр не нужен VK и не нужен нам для идентификации/учёта — вопроса в брифе быть не должно. Если VK требует параметр, а в брифе его нет — это дефект брифа.

Источник каждого параметра классифицируется:
- **бриф** — значение знает только клиент (его ниша, ссылка на сообщество, гео, бюджет);
- **оператор** — задаёт Анастасия при подтверждении (тонкий таргетинг, ставка);
- **дефолт** — берётся из шаблона цели (плейсменты, расписание по умолчанию).

**Вопросами брифа становятся только параметры с источником «бриф».**

## 1. Процедура разработки брифа (по шагам)

### Шаг 1. Вычитать актуальную документацию VK Ads
Источник истины — официальная документация и Swagger VK Рекламы (`target.vk.ru/help`, `target.vk.ru/doc`, ads.vk.ru). Выписать в таблицу (раздел 3) параметры по четырём уровням:
1. **Кабинет / рекламный аккаунт** — что нужно для создания аккаунта клиента (агентский доступ, название, объект рекламы).
2. **Кампания** — объект рекламирования + цель (по нашим 4 целям).
3. **Группа объявлений** — таргетинг (гео, демография, интересы/поведение, ключевые фразы, аудитории, плейсменты, расписание, устройства), бюджет, ставка/стратегия.
4. **Объявление** — формат, медиа, текст, заголовок, кнопка, ссылки, хэштеги.

> ВАЖНО: точные имена полей API и лимиты НЕ замораживать в этот документ как окончательные — техтребования VK меняются несколько раз в год. Таблица в разделе 3 — стартовый проход; при разработке сверять с живым Swagger.

### Шаг 2. Сверить с живым рекламным кабинетом
После получения доступа от Анастасии (кабинет VK / API-ключи):
- какие поля **обязательны** для запуска, какие опциональны;
- какие справочники отдаёт VK (список интересов, гео-справочник, список целей объекта);
- какие значения имеют дефолты;
- ограничения по форматам креативов (раздел 4) в текущем интерфейсе.

### Шаг 3. Классифицировать параметры по источнику
Для каждого параметра из таблицы проставить: бриф / оператор / дефолт. Зафиксировать в колонке «Источник».

### Шаг 4. Выделить обязательные вопросы брифа
Из параметров с источником «бриф» отобрать те, **без которых нельзя запустить** (раздел 5.1 — обязательные), и опциональные (раздел 5.2). Обязательные делают форму валидной; без них система не должна позволять отправку.

### Шаг 5. Добавить служебные поля
- **Идентификация клиента** (раздел 6.1): email, телефон, Telegram-username — спрашиваем **все три**, т.к. система идентифицирует клиента по совпадению любого из них.
- **Поля под создание кабинета и объекта рекламы** (раздел 6.2): ссылка на сообщество / профиль (получатель рекламы), название кабинета/кампании, ниша.

### Шаг 6. Сформулировать вопросы на языке клиента
Технические термины перевести в понятные формулировки с подсказками и примерами (раздел 7). Клиент не знает, что такое «look-alike» или «объект рекламирования», — он знает свою нишу, аудиторию, бюджет.

### Шаг 7. Согласовать с Анастасией
Получить от неё готовую Google-форму или список вопросов, сверить с нашей таблицей, дополнить недостающим под создание кабинета. Зафиксировать финальную структуру.

### Шаг 8. Реализовать форму и чтение
- Форма: Google Forms (Блок 1). РФ-вариант (Яндекс) — платная доработка (раздел 8).
- Чтение: Google Sheets API (сервис-аккаунт) + n8n-поллинг новых ответов.

### Шаг 9. Маппинг ответов брифа → параметры VK
Составить таблицу соответствия «поле формы → параметр VK API» (раздел 9). Это вход для `services/mapping.py`.

## 3. Таблица параметров VK (стартовый проход, уточняется по живой документации)

### Уровень: Кабинет / объект рекламы
| Параметр | Тип | Источник | Обяз. | Как спросить клиента |
|---|---|---|---|---|
| Объект рекламы (сообщество / профиль VK) | ссылка/URL | бриф | да | «Ссылка на ваше сообщество или страницу ВК, которую рекламируем» |
| Ниша / тематика | текст | бриф | да | «Чем вы занимаетесь? Что рекламируем?» |
| Название кампании (внутреннее) | текст | оператор/дефолт | нет | генерируется из ниши + даты |

### Уровень: Кампания (цель)
| Параметр | Тип | Источник | Обяз. | Примечание |
|---|---|---|---|---|
| Цель (1 из 4) | enum | бриф/оператор | да | подписчики / сообщения / лид-форма / Senler |

### Уровень: Группа объявлений (таргетинг + бюджет)
| Параметр | Тип | Источник | Обяз. | Как спросить клиента |
|---|---|---|---|---|
| География | страна/город/радиус | бриф | да | «На какие города/регионы показывать рекламу?» |
| Пол | enum | бриф | нет | «Ваша аудитория — мужчины, женщины или все?» |
| Возраст | диапазон | бриф | нет | «Возраст вашей аудитории (от — до)?» |
| Интересы / поведение | список | бриф/оператор | нет | «Чем интересуется ваша аудитория?» (свободно, оператор сопоставит со списком VK) |
| Ключевые фразы | список | бриф (опц.) | нет | «По каким запросам вас ищут?» (период поиска 1–30 дней) |
| Аудитории (ретаргетинг/look-alike) | список ID | оператор | нет | вне MVP по умолчанию |
| Плейсменты | список | дефолт | нет | по умолчанию авто |
| Расписание показов | расписание | дефолт/оператор | нет | по умолчанию круглосуточно |
| Устройства | enum | дефолт | нет | по умолчанию все |
| Бюджет (дневной / общий) | число | бриф | да | «Бюджет на рекламу (в день или всего)?» |
| Ставка / стратегия | число/enum | дефолт/оператор | нет | CPC/CPM; пополнение баланса — на стороне клиента/Анастасии |

### Уровень: Объявление (креатив) — см. раздел 4
| Параметр | Тип | Источник | Обяз. | Примечание |
|---|---|---|---|---|
| Формат | enum | оператор/дефолт | да | зависит от цели и медиа |
| Медиа (изображение / видео) | файл(ы) | оператор | да | принимается интерактивно в боте |
| Текст объявления | текст | бриф/оператор | да | лимиты зависят от формата |
| Заголовок | текст | бриф/оператор | зависит | для карусели/записи с кнопкой |
| Кнопка (CTA) | enum | дефолт/оператор | зависит | из списка VK |
| Хэштеги | текст | оператор (опц.) | нет | не во всех форматах допустимы |

## 4. Креативы: форматы и интерактивный приём

> Значения ниже — ориентиры из документации VK на момент написания. **Перед реализацией сверить с актуальными требованиями в интерфейсе VK Рекламы** — они меняются несколько раз в год. Брать актуальные программно/из справки, не хардкодить как константы без возможности обновления.

### 4.1. Опорные форматы и лимиты (ориентир)
- **Универсальная запись:** до 10 изображений, 1 видео и 1 сниппет; текст до 16 384 символов; минимальная ставка от 1 рубля.
- **Запись с кнопкой:** основной текст до 220 символов, не более двух переносов строк, без ссылок и хэштегов; текст сниппета до 80 знаков.
- **Карусель:** от 3 до 6 слайдов (для товаров — до 20); изображения 600×600; заголовок слайда до 40 знаков, описание до 50.
- **Изображения:** форматы JPG, PNG, GIF, TIF; соотношения 1:1, 4:5, 16:9; рекомендуемо от 1080×1080 (квадрат) и 1080×1920 (вертикаль); текст на изображении — не более 50% площади.
- **Видео:** соотношение 16:9, хронометраж 5–30 секунд; поддерживаются MP4, MOV, AVI и др.

### 4.2. Интерактивный приём креатива в боте
Бот ведёт оператора по шагам, опираясь на ограничения выбранного формата:
1. Определить тип материала: одно фото / несколько фото (карусель) / видео.
2. Принять медиа **по одному сообщению**, сохранить на сервере, запомнить порядок.
3. Спросить: «Добавить текст объявления?» — если да, принять текст и проверить лимит формата.
4. Спросить: «Добавить хэштеги?» — но **только если формат их допускает** (например, в записи с кнопкой хэштеги и ссылки запрещены — тогда шаг пропустить и сообщить почему).
5. Валидировать собранный креатив против актуальных требований формата (размеры, длина текста, число слайдов).
6. Показать итог и передать в общую сводку подтверждения (см. `PROJECT.md`, 4.1.6).

**Файлы:** `bot/handlers/creative.py`, `services/creative_store.py`, `services/creative_validate.py`.

## 5. Поля брифа

### 5.1. Обязательные (без них запуск невозможен)
- Объект рекламы (ссылка на сообщество / профиль).
- Ниша / что рекламируем.
- Цель (или определяется оператором, если клиент не выбирает).
- География.
- Бюджет.
- Медиа-креатив (принимается оператором в боте, но факт его наличия обязателен).
- Контакты идентификации (см. 6.1).

### 5.2. Опциональные
- Пол, возраст, интересы, ключевые фразы, заголовок, пожелания по тексту/CTA.

## 6. Служебные поля

### 6.1. Идентификация клиента (спрашиваем все три)
- Email.
- Телефон.
- Telegram-username.

Система идентифицирует клиента по совпадению **любого** из трёх. Совпало — привязываем бриф к существующему клиенту (новая кампания или новый кабинет в зависимости от объекта рекламы). Не совпало — заводим нового клиента.

### 6.2. Поля под создание кабинета
- Ссылка на объект рекламы (сообщество / профиль) — он же определяет «новый кабинет vs новая кампания для существующего».
- Название (генерируется, при необходимости правится оператором).

## 8. Развилка по персональным данным (152-ФЗ)

Бриф содержит контакты клиента (ПДн). Хранение ПДн граждан РФ должно происходить на серверах в РФ.
- **Блок 1 (по умолчанию):** Google Forms → Google Sheets API. Подходит, когда клиенты вне РФ либо как стартовый вариант showcase.
- **РФ-вариант (платная доработка):** Яндекс Формы или чтение Яндекс-почты (российская инфраструктура). Открытые вопросы для доработки: бесплатность API Яндекс Форм / чтения Яндекс-почты, необходимость платного аккаунта. Браузерная автоматизация Яндекс-почты возможна, но нестабильна (зависит от вёрстки) — закладывать только как крайний вариант.

В коде слой чтения брифа абстрагировать (`BriefSource` интерфейс с реализациями `GoogleFormsSource`, в будущем `YandexFormsSource`), чтобы добавление РФ-источника не ломало ядро.

## 9. Маппинг «поле брифа → параметр VK API»

Финальный артефакт этапа — таблица соответствия, вход для `services/mapping.py`:

| Поле брифа | Параметр VK (уровень) | Преобразование |
|---|---|---|
| Ссылка на сообщество | объект рекламы (кампания) | резолв ID сообщества |
| Цель | цель кампании | enum → код VK |
| География | гео-таргетинг (группа) | резолв в гео-ID VK |
| Пол/возраст | демография (группа) | прямое сопоставление |
| Интересы (текст) | интересы (группа) | сопоставление со справочником VK (оператор подтверждает) |
| Ключевые фразы | ключевые фразы (группа) | список строк |
| Бюджет | бюджет (группа) | число |
| Текст/медиа | объявление | сборка креатива |

Заполняется по итогам Шагов 1–2 (после сверки с живым API).

## 10. Открытые вопросы

1. **Агентский статус ИП в VK** — определяет, можем ли создавать кабинеты клиентов через VK API; уточнить у Анастасии (см. `DECISIONS.md`).
2. **Готовая форма от Анастасии** — получить её Google-форму или список вопросов; сверить и дополнить.
3. **Справочники VK** (интересы, гео) — получить через API при наличии доступа.
4. **РФ-форма (Яндекс)** — условия подключения; объём платной доработки.
