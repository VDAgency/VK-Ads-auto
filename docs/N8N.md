# N8N.md — правила работы с n8n (ОБЯЗАТЕЛЬНО перед действиями с n8n)

> **Жёсткое правило для агента.** Перед любыми действиями с n8n (воркфлоу, креды, деплой, импорт/экспорт) перечитай этот файл. n8n — отдельная и нетривиальная ветка; не действуй по памяти.

---

## 0. Две среды n8n (не путать)

| Среда | Что | Как ставится |
|---|---|---|
| **Локальная (разработка)** | простой официальный n8n, 1 контейнер + Postgres | минимальный `docker-compose.yml` в WSL2 (Docker рабочий) |
| **Прод (сервер)** | n8n в минимальной конфигурации платформы | установщик `kossakovsky/n8n-install`, выбрать ТОЛЬКО n8n + ядро (Caddy/Postgres/Redis) |

- Воркфлоу **строим и правим локально**, затем через git выкатываем на сервер.
- Прод НЕ источник правды. Источник правды — **git**.
- Локальный установщик kossakovsky не используем (требует публичный VPS + домен + wildcard-DNS; на локали Caddy не выдаст HTTPS).

---

## 1. Воркфлоу = код в git

- Лежат в репозитории: `n8n/workflows/*.json` (каждый воркфлоу — отдельный файл).
- **В JSON воркфлоу НЕТ секретов** — только ссылки на креды по имени.
- Цикл (как у кода): правим локально → экспорт в файлы → commit → push → на сервере импорт.

### Экспорт (локально, после правок в UI)
```bash
docker compose exec n8n n8n export:workflow --all --separate --output=/data/workflows
# затем скопировать JSON в репозиторий n8n/workflows/ и закоммитить
```

### Импорт (на сервере, при деплое)
```bash
docker compose exec n8n n8n import:workflow --separate --input=/data/workflows
```
> `make import` из установщика kossakovsky — это про их community-воркфлоу, НЕ про наши. Наши воркфлоу импортируем CLI n8n, как выше.

---

## 2. Креды (ключи) — мимо git, всегда

- Креды (Google, Senler, и т. п.) хранятся в **зашифрованной БД n8n**, НЕ в JSON воркфлоу.
- Вводятся **руками в UI n8n** — отдельно на локальном и отдельно на серверном экземпляре.
- **Имена credential на локали и на сервере должны СОВПАДАТЬ** — иначе импортированный воркфлоу не подхватит ключ.
- Когда нужно ввести креды на сервере — агент сообщает разработчику, что и под каким именем вписать; сам ключи в код/`.env`/git не помещает.

### Сохранность кред (обязательно)
- `N8N_ENCRYPTION_KEY` — задать явно, держать одинаковым на каждом экземпляре, хранить как секрет (НЕ в git).
- Подключить постоянный **volume** под данные n8n — иначе при пересоздании контейнера креды протухнут и их придётся вбивать заново.

## 3. Наша система ↔ n8n

- Связь только по **REST API** n8n.
- `N8N_API_URL` + `N8N_API_KEY` — в серверном `.env` (см. `docs/DEPLOY.md`).
- API-ключ создаётся **внутри UI n8n**: Settings → n8n API → Create API key (бесплатно, self-hosted Community). Появляется после развёртывания n8n.
- На Блоке 1 поллинг брифов и уведомления ведём через REST n8n (не через MCP).

---

## 4. MCP n8n — опциональный помощник, не основной путь

- `n8n-mcp` (czlonkowski) — для проектирования/валидации/чтения воркфлоу агентом.
- В текущей среде **нестабилен** (VPN + TLS-перехват ломают stdio-хэндшейк) → на него НЕ полагаемся.
- Без `N8N_API_KEY` работает в режиме документации/валидации; с ключом может смотреть/править живой n8n.
- **Изменения продакшна идут через git + import, а не через запись MCP в живой n8n.** Если MCP отвалился — сообщить, оценить, продолжить через CLI/REST.

---

## 5. База данных (определяется при настройке сервера)

- Установщик kossakovsky поднимает свои Postgres и Redis для платформы.
- **Открытый вопрос (решить при настройке сервера):** наш код использует тот же Postgres (отдельная БД внутри него) или поднимаем свой инстанс. Аналогично Redis.
- До решения — НЕ хардкодить подключение; вынести в `.env`. Зафиксировать выбор здесь после настройки сервера.

---

## 6. Порядок при старте n8n-части

1. Локально поднять простой n8n (1 контейнер + Postgres) в WSL2.
2. Построить воркфлоу в UI, ввести локальные креды.
3. Экспорт → `n8n/workflows/*.json` → commit → push.
4. На сервере развернуть n8n установщиком (минимальный набор), задать `N8N_ENCRYPTION_KEY` + volume.
5. Импорт воркфлоу на сервере; ввести креды в серверном UI под теми же именами.
6. Создать `N8N_API_KEY` в UI сервера → внести в серверный `.env` → включить связь нашего кода по REST.
