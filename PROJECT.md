# PROJECT.md — Система автоматизации запуска рекламы в VK Ads

**Роли:** Заказчик · Оператор (единственный пользователь системы) · Исполнитель (разработка и поддержка).
**Договор:** разработка сервиса автоматизации запуска рекламы в VK Ads через Telegram-бот · два блока.

> Этот документ — «что и как» по проекту. Метод сборки брифа вынесен в `BRIEF_SPEC.md`, последовательность работ — в `ROADMAP.md`, история и обоснования решений — в `DECISIONS.md`.

---

## 1. Обзор и цели

Система снимает с таргетолога рутину запуска рекламы в VK Ads: оператор ставит задачу в Telegram-боте, система принимает бриф клиента, раскладывает его в настройки кампании, по подтверждению создаёт рекламный кабинет и запускает рекламу на одну из 4 целей, затем собирает статистику. Ориентир: от постановки задачи до запуска — около 10 минут.

**Цели проекта:**
- сократить ручной труд при запуске рекламы;
- единый интерфейс запуска и контроля в Telegram;
- заложить масштабируемую архитектуру (мультикабинетность, адаптер интеграции, задел под будущие расширения).

**Сквозные принципы:**
1. **Адаптерность интеграций (исходящая)** — путь интеграции (kotbot / прямой VK API) не зашит в ядро, а скрыт за адаптером с переключателем. Этот же слой — точка расширения на другие площадки в будущем.
2. **Российская инфраструктура и аккуратность с ПДн** (152-ФЗ) — хранение на сервере в РФ, развилка по формам сбора брифа.
3. **Поэтапность** — Блок 1 (Telegram-бот) самодостаточен, Блок 2 (веб + клиентский портал) надстраивается сверху.
4. **API-first / headless-ядро (входящая абстракция)** — вся бизнес-логика живёт только в ядре; бот, веб, mini-app и будущие клиенты — тонкие клиенты над версионированным внутренним API. В интерфейсах бизнес-логики нет. Добавить новый фронт = новый тонкий клиент без изменений ядра.
5. **Готовность к мульти-тенантности** — строим однотенантно (оператор = «тенант №1»), но со швами под many-tenant: `account_id` на всех таблицах, изоляция строк по тенанту, конфиг интеграций per-account. Полноценный мульти-тенант-продукт в текущем объёме не реализуется — заложены только архитектурные швы (см. разделы 5, 9).

## 2. Роли и пользователи

- **Оператор** — единственный пользователь системы. Все задачи, подтверждения и отчёты проходят через него.
- **Клиент** — заказчик рекламы у оператора. Может быть частным лицом, а может субагентством со своими клиентами. Получает ссылку на бриф; на Блоке 2 — клиентский мини-кабинет.
- **Получатель рекламы** — объект, на который льётся реклама: сообщество или профиль VK.
- **Администратор системы (Исполнитель)** — деплой, поддержка, доработки.

**Иерархия данных:**
```
Оператор
└── Клиенты (включая самого оператора как «клиента» для своей рекламы)
    └── Кабинеты (1 клиент : N кабинетов)
        └── Объект рекламы (сообщество / профиль VK)
```

## 3. Сценарии end-to-end

### Сценарий A — запуск от брифа до старта
1. Оператор в боте жмёт «отправить бриф» и вводит контакт клиента (почта / телефон / Telegram-username — тип определяется автоматически).
2. Система отправляет клиенту ссылку на бриф с коротким сопроводительным текстом и поднимает флаг «ждём бриф от клиента X».
3. Клиент заполняет форму.
4. Система по расписанию обнаруживает новый ответ → оператору приходит уведомление «клиент X заполнил».
5. Система делает алгоритмическую раскладку полей брифа в параметры кампании.
6. Бот интерактивно принимает креатив (медиа + текст + при необходимости хэштеги).
7. Бот показывает итоговую сводку (клиент, объект, цель, параметры, креатив) с кнопками **«Подтвердить / Исправить»**.
8. После подтверждения — уход в выбранный канал (kotbot / VK API).
9. Через 1–2 минуты — «кабинет создан, кампания запущена, добавлено в отслеживание».

### Сценарий B — правки
При нажатии «Исправить» оператор отправляет правки в формате `номер.значение`, можно несколько за одно сообщение через абзац:
```
1. подписчики
5. 30000
```
Система применяет правки и присылает обновлённую сводку — до финального «OK».

### Сценарий C — отчётность
- Ежедневный дайджест в 9:00 по всем кабинетам (коротко).
- По запросу: оператор открывает конкретную кампанию → триггер «обновить метрики» → ответ «метрики обновлены, актуальные значения».

### Сценарий D (Блок 2)
Клиент по реферальной ссылке сам заполняет бриф → заводится в системе → получает доступ к мини-кабинету (magic-link, смена пароля) → видит мини-отчёт по своим целям (без расхода) → может пригласить другого клиента и получить скидку; оператору — уведомление «X привёл Y, скидка Z».

## 4. Разбор ТЗ построчно

Формат: **что → почему так → технологии → файлы → результат для пользователя.**

### Блок 1 — Telegram-бот (MVP)

#### 4.1.1. Telegram-бот: постановка задачи и загрузка креатива
- **Что:** интерфейс оператора на aiogram 3 с пошаговым диалогом (FSM).
- **Почему:** Telegram — привычная среда, мгновенный старт без веба.
- **Технологии:** aiogram 3, FSM-состояния, Redis для хранения состояний.
- **Файлы:** `bot/main.py`, `bot/handlers/*`, `bot/states.py`, `bot/keyboards.py`.
- **Результат:** оператор управляет всем запуском из чата.

#### 4.1.2. Приём и разбор брифа
- **Что:** оператор отправляет ссылку на форму контакту клиента; клиент заполняет; система по расписанию читает ответы и разбирает их в структурированные поля. Метод сборки самого брифа — в `BRIEF_SPEC.md`.
- **Почему:** убирает ручной сбор данных; чтение связанной таблицы по API надёжнее парсинга письма.
- **Технологии:** Google Forms → Google Sheets API (сервис-аккаунт); n8n — поллинг новых ответов и уведомления; алгоритмический парсер полей.
- **Развилка ПДн:** Google хранит данные за рубежом — для клиентов из РФ это риск по 152-ФЗ. На Блоке 1 используем Google (работает сразу); Яндекс-вариант (Яндекс Формы или чтение Яндекс-почты) выносим в платную доработку.
- **Файлы:** `integrations/google_forms.py`, `services/brief_parser.py`, n8n-воркфлоу `brief_poller`.
- **Результат:** заполненный бриф автоматически попадает в систему, оператор получает уведомление.

#### 4.1.3. Создание рекламного кабинета (адаптер)
- **Что:** по подтверждённой задаче система создаёт кабинет клиента и кампанию через выбранный канал.
- **Почему:** это и есть автоматизация, ради которой заказана услуга.
- **Технологии:** адаптер `PlatformAdapter` с реализациями `VkApiAdapter` (прямой VK Ads API) и `KotbotAdapter` (Playwright-автоматизация интерфейса, т.к. API у kotbot нет); переключатель (дефолт + ручной) и health-check с фолбэком.
- **Файлы:** `integrations/adapter.py`, `integrations/vk_api.py`, `integrations/kotbot_playwright.py`.
- **Результат:** кабинет и кампания создаются без ручной работы в интерфейсе.
- **Флаг:** создание кабинетов клиентов через VK API доступно агентскому доступу на юрлицо — нужно уточнить агентский статус ИП (см. раздел 10).

#### 4.1.4. Мультикабинетность
- **Что:** система ведёт много клиентов и много кабинетов под единственным оператором.
- **Почему:** у оператора может быть 100–300 клиентов, у клиента — несколько кабинетов.
- **Технологии:** PostgreSQL, схема с иерархией Клиент 1:N Кабинет.
- **Файлы:** `db/models.py`, `db/migrations/*`.
- **Результат:** единое окно по всем клиентам и кабинетам.

#### 4.1.5. Алгоритмическая раскладка
- **Что:** преобразование полей брифа в параметры кампании по правилам/шаблонам (без ИИ).
- **Почему:** предсказуемо, дёшево, без затрат на LLM на Блоке 1.
- **Технологии:** правила-маппинг (поле брифа → параметр VK), шаблоны на каждую из 4 целей; значения из брифа / ввода оператора / дефолта. Перечень параметров — в `BRIEF_SPEC.md`.
- **Файлы:** `services/mapping.py`, `services/templates/*`.
- **Результат:** из брифа получается готовый черновик настроек кампании.

#### 4.1.6. Подтверждение и правка (`номер.значение`)
- **Что:** сводка с кнопками «Подтвердить / Исправить»; правки по номерам, несколько за сообщение.
- **Почему:** быстрый контроль без сложных форм в чате.
- **Технологии:** aiogram FSM, парсер правок (regex `^(\d+)\.\s*(.+)$` построчно), карта «номер → параметр».
- **Файлы:** `bot/handlers/confirm.py`, `services/edit_parser.py`.
- **Результат:** оператор за секунды корректирует и подтверждает запуск.

#### 4.1.7. Запуск на 4 цели
- **Что:** поддержка целей подписчики / сообщения / лид-форма / заявка через Senler.
- **Почему:** закрывает основные задачи таргетолога.
- **Маппинг на VK:**
  - подписчики — объект «Сообщество», цель «Подписаться»;
  - сообщения — объект «Сообщество», цель «Написать сообщение»;
  - лид-форма — объект «Лид-форма»;
  - заявка через Senler — через цель «Написать сообщение» (заявку ловит чат-бот Senler, подключённый к сообществу). Отдельной нативной цели «Senler» в VK нет.
- **Технологии:** профили целей в адаптере, Senler API.
- **Файлы:** `services/goals.py`, `integrations/senler.py`.
- **Результат:** запуск под нужную цель в один поток.

#### 4.1.8. Интерактивный приём креатива
- **Что:** бот пошагово принимает креатив (картинка / карусель картинок / видео) и текст, спрашивая по ходу «добавить хэштеги? добавить текст?».
- **Почему:** форматы и лимиты VK сложные и меняются — оператора надо вести.
- **Технологии:** aiogram (приём media по одному, сохранение на сервере), валидация по актуальным требованиям VK. Детали форматов — в `BRIEF_SPEC.md`, раздел «Креативы».
- **Файлы:** `bot/handlers/creative.py`, `services/creative_store.py`, `services/creative_validate.py`.
- **Результат:** корректно собранный креатив, готовый к загрузке в кабинет.

#### 4.1.9. Сбор статистики и отчёты
- **Что:** ежедневный дайджест 9:00 + обновление по запросу при входе в кампанию.
- **Почему:** контроль без ручного захода в кабинеты.
- **Технологии:** планировщик (APScheduler / n8n), запрос статистики через адаптер, хранение срезов в БД.
- **Метрики:** показы, расход, результат по цели, CPL/CPC, CTR.
- **Файлы:** `services/stats.py`, `bot/handlers/reports.py`, воркфлоу `daily_digest`.
- **Результат:** оператор видит актуальные метрики по кампании и сводно.

### Блок 2 — Веб-интерфейс и клиентский портал

#### 4.2.1. Веб-интерфейс с авторизацией
- **Что:** веб-кабинет на своём домене.
- **Технологии:** Next.js + FastAPI-бэкенд, на том же Beget VPS.
- **Файлы:** `web/*`.
- **Результат:** доступ к системе из браузера.

#### 4.2.2. Клиентский мини-кабинет
- **Что:** профиль клиента, мини-отчёт по его целям (показы, подписчики, лиды) **без расхода** (не светим маржу), доступ по факту брифа.
- **Технологии:** magic-link на контакт клиента → вход → смена пароля.
- **Файлы:** `web/client/*`, `services/auth_magiclink.py`.
- **Результат:** клиент сам смотрит результат, не дёргая оператора.

#### 4.2.3. Одноуровневая реферальная система
- **Что:** реф-ссылка клиента → приведённый клиент → скидка на 1 следующий месяц (размер согласуем позже) → уведомление оператору; ведём таблицу учёта скидок (задел под будущую автоматизацию платежей).
- **Технологии:** трекинг реф-кодов, таблица скидок.
- **Файлы:** `services/referral.py`, `db/models.py` (Referral, Discount).
- **Поля таблицы скидок:** реферер, приведённый клиент, % скидки, месяц действия, статус (применена/нет), дата.
- **Результат:** рост по рекомендациям + прозрачный учёт скидок.

#### 4.2.4. Самостоятельная отправка брифа по реф-ссылке
- **Что:** клиент сам открывает бриф по ссылке и отправляет; оператору — уведомление о новом клиенте.
- **Технологии:** те же Google Forms + трекинг источника.
- **Файлы:** `web/referral/*`.
- **Результат:** поток новых клиентов без участия оператора.

## 5. Архитектура

Система построена как **headless-ядро с тонкими клиентами** и двумя слоями абстракции — входящим (фронты → ядро) и исходящим (ядро → рекламные площадки). Эти два слоя не путать.

**Компоненты:** ядро (FastAPI: бизнес-логика + внутренний API) · PostgreSQL · Redis (состояния / кеш / очереди) · n8n (поллинг Google Forms / почты + уведомления) · слой интеграции (исходящий адаптер) · тонкие клиенты: Telegram-бот (aiogram 3), веб Next.js (Блок 2), в будущем — Telegram mini-app, мобильные приложения. Всё в Docker Compose на Beget VPS.

**Входящий слой — API-first / headless.** Вся бизнес-логика живёт только в ядре. Ядро отдаёт версионированный внутренний REST API (`/api/v1/...`) — единый контракт для всех интерфейсов. Каждый интерфейс — тонкий клиент: собирает ввод, рендерит результат, логики не содержит. Бот, деплоясь вместе с ядром, может звать сервисный слой in-process; веб и мобильные — по REST. **Инвариант: логика только в ядре, интерфейсы сменные.** Добавить новый фронт = новый тонкий клиент без изменений ядра.

**Исходящий слой — интеграционный адаптер.** Интерфейс `PlatformAdapter` (`create_cabinet()`, `create_campaign()`, `upload_creative()`, `launch()`, `get_stats()`) с реализациями `VkApiAdapter` и `KotbotAdapter`. Выбор канала — `IntegrationConfig` (дефолт + ручной переключатель). Health-check + фолбэк с обратной связью оператору. Этот же слой — точка расширения на другие площадки (Avito, OK и пр.) в будущем: новая реализация адаптера, ядро не трогаем.

**Готовность к мульти-тенантности.** Ядро строится однотенантно (оператор = тенант №1), но со швами: `account_id` на всех таблицах, скоупинг всех запросов по тенанту, конфиг интеграций (VK-ключи, токен бота, Google, Senler) — **per-account**, а не глобальные константы. Полноценный мульти-тенант-продукт в текущем объёме не реализуется (раздел 9).

**Поток данных:**
```
[Тонкие клиенты: бот / веб / mini-app] ⇄ Внутренний API ядра (FastAPI)
Ядро: разбор брифа, раскладка, бизнес-логика, изоляция по account_id
n8n → (поллинг Google Forms / почты) → ядро (новый бриф) → уведомление клиенту-интерфейсу
Ядро → PlatformAdapter (VK / kotbot) → VK Ads
Планировщик → статистика → ядро → клиент
```

## 6. Модель данных (PostgreSQL)

- **Account (Tenant)** — владелец инстанса данных; `account_id` присутствует на ВСЕХ таблицах, все запросы скоупятся по тенанту. Сейчас один тенант (оператор), но изоляция заложена с первого дня — это шов под мульти-тенант.
- **Operator** — принадлежит Account; пользователь, ставящий задачи.
- **Client** — принадлежит Operator; контакты (email, phone, tg_username) для идентификации; флаг «это сам оператор».
- **Cabinet** — принадлежит Client (1 Client : N Cabinet); привязка к каналу (VK / kotbot) и объекту рекламы.
- **AdObject** — сообщество / профиль VK (получатель рекламы).
- **Campaign** — принадлежит Cabinet; цель, статус, параметры раскладки.
- **Creative** — медиа (изображения / видео) + текст + хэштеги; привязан к Campaign.
- **Brief** — сырой и разобранный бриф; связь с Client (по совпадению любого из контактов) и Campaign.
- **Goal** — справочник 4 целей и их маппинга на объекты / цели VK.
- **Stat** — срезы метрик по Campaign / Cabinet с таймстампом.
- **Referral / Discount** — реферер, приведённый, %, месяц, статус, дата (Блок 2).
- **IntegrationConfig** — **per-account**: канал по умолчанию, ключи (VK client_id/secret, токен бота, Google, Senler), состояние health-check. Ключи привязаны к аккаунту, а не зашиты глобально — это ключевой шов под мульти-тенант (его дороже всего ретрофитить потом).

**Идентификация клиента:** при вводе контакта или приходе брифа сверяем email / phone / tg_username; совпадение → существующий клиент (новая кампания / кабинет по объекту рекламы), нет совпадений → новый клиент.

## 7. Стек и технологии

Python 3.11+, FastAPI (async-ядро), aiogram 3 (бот), PostgreSQL (данные), Redis (состояния / очереди), n8n self-hosted (поллинг / уведомления, чтобы оператор позже могла сама расширять простые сценарии), Playwright (автоматизация kotbot), Next.js (веб Блока 2), Google API (Forms / Sheets), Senler API, VK Ads API. Развёртывание — Docker Compose на **Beget VPS** (РФ, под ПДн).

## 8. Интеграции

**VK Ads (VK Реклама).** Доступ к API запрашивает владелец кабинета: «Профиль» → «Доступ к API» → «Запросить доступ», далее пара client_id / client_secret (secret виден ~10 минут). Кампания строится в три уровня: кампания (цель) → группа объявлений (таргетинг) → объявление (креатив). Таргетинг: география (вплоть до точки с радиусом до 500 м), демография (пол, возраст), интересы и поведение, аудиторные списки (ретаргетинг, look-alike), плейсменты, расписание показов, устройства, UTM-метки (авто), ключевые фразы (период поиска 1–30 дней). Оплата — CPC (за уникальный клик) или CPM (за 1000 показов).

**kotbot.** API нет → автоматизация веб-интерфейса через Playwright; путь хрупкий (зависит от вёрстки) → health-check и фолбэк на VK API.

**Google.** Forms → Sheets API через сервис-аккаунт; n8n следит за новыми ответами.

**Yandex (опционально, платно).** Яндекс Формы / Яндекс-почта для ПДн-сценария РФ — отдельная доработка (изучить бесплатность API / чтения почты; браузерная автоматизация почты возможна, но нестабильна).

**Senler.** API-ключ; цель «заявка через Senler» реализуется через «Написать сообщение» (бот Senler ловит обращение).

**Telegram.** aiogram 3, токен @BotFather; команды принимаются только от оператора по её Telegram ID.

## 9. Границы скоупа

**Вне скоупа:** генерация креативов (делает оператор); маркировка (на стороне VK — проверка на соответствие ЕРИР/ОРД выполняется на этапе модерации в кабинете); приём платежей и автопополнение баланса; мультиплатформа (Avito / OK и пр.); AI-раскладка; мониторинг бюджета (автонапоминания / сводки — будущая платная доработка); Яндекс-интеграция (платная доработка). Контроль бюджета — задача оператора.

**Расширения за рамками текущего объёма** (мультиплатформа и пр.) в этом документе не описываются — заложены только архитектурные швы (разделы 1, 5, 6).

## 10. Открытые вопросы и флаги

1. **Агентский статус ИП в VK** — уточнить у оператора; от него зависит создание кабинетов клиентов через VK API. Подготовим письмо с условиями регистрации агентства.
2. **Поля брифа** — финализировать по `BRIEF_SPEC.md` и согласовать с оператором; получить её готовую Google-форму или список вопросов.
3. **Точные поля VK Ads API** — сверить с живым Swagger target.vk.ru при разработке (имена полей не выдумываем).
4. **Размер реферальной скидки** — согласовать с оператором.
5. **Техтребования к креативам VK** меняются несколько раз в год — брать актуальные из интерфейса / документации, не хардкодить.
6. **kotbot** — есть ли у них сейчас агентское создание кабинетов; иначе сразу прямой VK.
