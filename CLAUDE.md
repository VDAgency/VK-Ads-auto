# CLAUDE.md — VK-Ads-auto

> **Главная инструкция проекта. Claude Code читает её в начале КАЖДОЙ сессии и следует ей неукоснительно.** Она задаёт правила процесса и связывает документы.
>
> Приоритет при конфликте: **CLAUDE.md (правила процесса) → PROJECT.md (архитектура) → docs/ROADMAP.md (план).** Если код противоречит PROJECT.md — прав документ, код исправляется. Если действие противоречит CLAUDE.md — действие не выполняется.

Карта файлов (что читать и когда):
- `PROJECT.md` (корень) — что за система, архитектура, модель данных.
- `docs/ROADMAP.md` — план и порядок фаз, гейт качества; отмечать `- [x]` сразу.
- `docs/BRIEF_SPEC.md` — метод сборки брифа из параметров VK.
- `docs/DECISIONS.md` — почему так решено (+ «устарело»).
- `docs/DEPLOY.md` — **обязательно перед любым деплоем / SSH / ключами.**
- `docs/N8N.md` — **обязательно перед любыми действиями с n8n.**
- `docs/ДИЗАЙН-СИСТЕМА.md` — дизайн-конвейер (Блок 2, веб).

## 0. Как работать с проектом (первое в сессии)

1. Прочитать CLAUDE.md (этот файл).
2. Прочитать PROJECT.md — архитектура, модель данных, решения.
3. Открыть docs/ROADMAP.md — найти текущую незавершённую задачу (`- [ ]`).
4. **Перед действиями деплоя — перечитать docs/DEPLOY.md; перед действиями с n8n — docs/N8N.md.** Это касается и параллельных сессий. Не держать имена ключей, пути и шаги по памяти.
5. Работать строго по порядку фаз, не перепрыгивать без явного указания. После задачи — `- [x]`.

Язык: общение с пользователем (Вячеслав) — **русский**. **Перед каждым значимым шагом писать по-русски, что именно делается**, чтобы был виден ход работы. Код, команды, имена веток и коммиты — английский.

## 1. ГЛАВНЫЕ ЗАПРЕТЫ (нарушение недопустимо)

### 1.1. Секреты (репозиторий ПУБЛИЧНЫЙ — критично)
- **НИКОГДА** не коммитить секреты: ключи VK (client_id/secret), токен бота, ключи Google/Senler, `N8N_ENCRYPTION_KEY`, `N8N_API_KEY`, пароли БД, SSH-ключи, строки подключения.
- **НИКОГДА** не писать секреты в код, комментарии, документацию, логи, сообщения коммитов. Только `.env` (в `.gitignore`) и переменные окружения.
- **НИКОГДА** не выводить содержимое `.env`.
- Перед каждым коммитом проверять diff на секреты. Есть — остановиться, убрать, закрыть `.gitignore`.
- Данные клиентов оператора (контакты, брифы, ПДн) — конфиденциальны; не отправлять во внешние сервисы сверх необходимого.

### 1.2. Git-процесс
- **НИКОГДА** не коммитить напрямую в `main`. Вся работа — через отдельные ветки и pull request.
- **НИКОГДА** не пушить код, не прошедший тесты. Сначала зелёные проверки — потом коммит/push.
- **НИКОГДА** не делать `git push --force` в `main` / общие ветки.
- **НИКОГДА** не коммитить мусор: `__pycache__`, `.venv`, `node_modules`, `.env`, дампы, временные файлы.

### 1.3. Архитектурные инварианты
- **Headless-ядро.** Каналы (Telegram-бот, веб, mini-app, будущий мобайл) ходят **только через внутренний API ядра (FastAPI)**. Бизнес-логика — только в `core/`/`services/`. **НИКОГДА** не давать каналам прямой доступ к БД и не класть SQL в обработчики каналов/роутеры — только через слой репозиториев/сервисов.
- **Исходящее — только через `PlatformAdapter`** (VK API / kotbot). Ядро не знает про конкретные площадки.
- **Мульти-тенант-готовность.** `account_id` на ВСЕХ таблицах; все запросы скоупятся по тенанту; конфиг интеграций — per-account, не глобальные ключи.
- **ПДн — только на серверах РФ** (152-ФЗ).

### 1.4. Скоуп
- **НИКОГДА** не расширять скоуп самовольно. Вне MVP: AI-раскладка, приём платежей/автопополнение, мультиплатформа, мониторинг бюджета, Яндекс-формы (платная доработка позже). Кажется, что задача требует этого — остановиться и спросить.
- **«Будущие треки»** (будущие расширения) — НЕ трогать, пока не сдан основной договор (docs/ROADMAP.md → «Будущие треки»).
- **Боевое создание кабинетов VK** — только после подтверждения агентского статуса ИП (открытый вопрос, PROJECT.md/DECISIONS.md).

## 2. ОБЯЗАТЕЛЬНЫЙ ЦИКЛ ШАГА

Каждая задача из ROADMAP выполняется строго по циклу:
1. **Объявить шаг** по-русски: что и зачем.
2. **Ветка** (не `main`): `feature/<...>`, `fix/<...>`, `chore/<...>`.
3. **Код** по конвенциям (раздел 4).
4. **Тесты** на этот код.
5. **Гейт качества локально:** `ruff check .` + `ruff format --check .` → `mypy .` → `pytest -q`. Фронт (Блок 2): `eslint` + `prettier --check` + `tsc --noEmit`. Всё зелёное.
6. **Красное** — чинить, не двигаться дальше.
7. **Атомарный коммит** (англ., `<type>: ...`: feat/fix/test/docs/chore/refactor).
8. **Push** в ветку → pull request в `main`.
9. **CI + Aikido (через GitHub-интеграцию) зелёные** перед merge.
10. **Merge в `main` → автодеплой** (механика — docs/DEPLOY.md).
11. **Отметить** `- [x]` в docs/ROADMAP.md.

Принцип: один шаг = одна откатываемая версия. Не копить большие необкоммиченные изменения.

### 2a. Завершение фазы (обязательно: commit → push → deploy)
В конце КАЖДОЙ фазы: гейт качества зелёный → commit → push → PR → CI+Aikido зелёные → **merge в `main` → автодеплой на сервер** → проверить `/health` → отметить чекбоксы фазы. Фаза без выкатки и проверки на сервере не считается закрытой.

## 4. КОНВЕНЦИИ КОДА

- Код и идентификаторы — английский; комментарии и пояснения шагов — русский.
- **Python 3.11** через **uv** (системный — 3.13, учитывать).

- Логирование — через модуль логирования, не `print`; структурированно.
- Конфигурация — только через env/`.env`, читается из `config/`. Никаких хардкод-секретов/окружения.
- Миграции БД — только через **Alembic**, не править схему руками в обход.
- Доступ к данным — через репозитории/сервисы ядра; обработчики каналов и роутеры без прямого SQL.
- Async для I/O и поллинга; фоновые воркеры — отдельные процессы, состояние/курсор в Redis.
- Воркфлоу n8n — JSON в `n8n/workflows/` (docs/N8N.md), без секретов.

## 5. ОРКЕСТРАЦИЯ СУБАГЕНТОВ, СКИЛЛОВ И MCP

> Реальные инструменты из инвентаризации окружения. Использовать именно их; не выдумывать отсутствующие.
>
> **Окружение и устойчивость:** Windows 10 + MSYS/MINGW64; Docker через **WSL2 (Linux) — рабочий**. В среде VPN + TLS-перехват (`NODE_TLS_REJECT_UNAUTHORIZED=0`) → MCP иногда отваливаются. Если MCP упал: **сообщить, оценить, блокирует ли шаг; если нет — продолжить через CLI/REST, не вставать колом.** Машина небыстрая — долгие операции это норма, не считать зависанием.

### 5.1. MCP-серверы — когда какой
- **`context7`** — актуальная документация библиотек (aiogram 3, FastAPI, Next.js, VK Ads API, Playwright) ПЕРЕД кодом под незнакомый/обновлённый API. Не по памяти.
- **`postgres`** — схема/данные/отладка запросов ПОСЛЕ поднятия БД; до этого статус Failed — норма, не «чинить».
- **`github`** — репозиторий, ветки, PR, статус CI/Actions, GitHub Secrets.
- **`aikido`** — безопасность; **только через GitHub-интеграцию** (локальный скан «по ссылке» ломается из-за VPN — не использовать).
- **`n8n-mcp`** — опционально, проектирование/валидация воркфлоу; нестабилен в этой среде → не полагаться (детали docs/N8N.md). Изменения прода — через git, не записью MCP в живой n8n.
- **`chrome-devtools`** — отладка/perf/a11y в браузере (Блок 2; инспекция потока kotbot).
- **(Блок 2) `shadcn`, `playwright`** — ставятся per-project на вёрстке (docs/ДИЗАЙН-СИСТЕМА.md). Playwright нужен и под автоматизацию kotbot.
- `excel` — проекту не нужен (бриф через Google Sheets).

### 5.2. Скиллы — когда какой
- **Планирование:** `superpowers:writing-plans` (разбить фазу на шаги), `superpowers:executing-plans`, `superpowers:verification-before-completion` (сверка с DoD, раздел 8).
- **Разработка/отладка (критичная логика — адаптер, парсер брифа, раскладка, правки `номер.значение`):** `superpowers:test-driven-development`, `superpowers:systematic-debugging` (корневая причина ПЕРЕД правкой).
- **Git:** `commit-commands:commit`, `commit-commands:commit-push-pr`, `commit-commands:clean_gone`.
- **Ревью перед merge:** `code-review:code-review` + `coderabbit:coderabbit-review` / `coderabbit:autofix`.
- **Архитектура/разведка фич:** `feature-dev:*`. **Упрощение кода:** `code-simplifier`.
- **Дизайн (Блок 2):** базовый `frontend-design` → апгрейд Impeccable (docs/ДИЗАЙН-СИСТЕМА.md).
- **Документы:** `document-skills:*` по необходимости.
- **Поиск по коду:** `ripgrep` (rg) — текстовый; `ast-grep` — структурный (рефакторинг/паттерны).
- **Сопровождение этого файла:** `claude-md-management:*` — только по явному запросу, не самовольно.

### 5.3. Параллельная работа — параллелить МАКСИМАЛЬНО
Директива: основной агент **агрессивно делегирует независимые подзадачи параллельным субагентам**, чтобы не тормозить основную разработку.
- `superpowers:dispatching-parallel-agents` — запуск нескольких субагентов на независимые подзадачи.
- `superpowers:subagent-driven-development` — делегирование подзадач.
- `superpowers:using-git-worktrees` — изоляция параллельной работы по worktree, чтобы агенты не конфликтовали на файлах.

Логические роли субагентов: **бэкенд** (ядро/API/интеграции), **тестировщик** (pytest, покрытие критичного), **ревьюер** (конвенции + инварианты 1.3), **безопасность** (секреты, 152-ФЗ, изоляция данных).

Параллельные сессии: каждая в своей ветке/worktree; перед действиями деплоя/n8n — перечитывают docs/DEPLOY.md и docs/N8N.md.

### 5.4. Безопасность перед PR в `main` (обязательно)
- Прогнать Aikido (через GitHub-интеграцию) на утечку секретов и уязвимости — машинная страховка к запрету 1.1.
- Вручную проверить, что в diff нет `.env`, ключей, паролей, токенов.

### 5.5. Что параллелить, что строго последовательно
**Параллельно** (независимые артефакты): документация к готовому модулю + следующий модуль; тестовые моки + код, который они проверяют; исследование context7 по нескольким библиотекам сразу.
**Последовательно** (зависимость/риск):
- Схема БД (Alembic) → код поверх. Сначала стабильная схема.
- Код → тесты → ревью → коммит (внутри шага порядок не нарушать).
- Ядро/внутренний API → каналы (бот ходит через ядро, значит ядро раньше).
- `PlatformAdapter` → запуск кампаний (запуск идёт через адаптер).
- Миграции БД — по одной, не пачкой конкурирующих.
- Боевое подключение VK/kotbot — только после подтверждения (агентский статус ИП), не «на всякий случай» в обход блокировки.
**Правило:** два субагента не правят один файл/одну миграцию одновременно. Риск коллизии → последовательно.

### 5.6. Синхронизация веток перед новой фазой (обязательно)
- `git branch -a` + `gh pr list` — выявить чужие ветки и открытые PR параллельных сессий.
- Смержить готовое (зелёный CI) ПЕРЕД фазой, в правильном порядке (сначала то, от чего зависят другие), проверяя CI на каждом мерже.
- Обновить свою ветку от свежего `main`.
- Конфликты по docs/ROADMAP.md — **ОБЪЕДИНЯТЬ `[x]`, не затирать** (и свои, и чужие отметки).
- Новую фазу стартовать только от актуального `main`.

## 6. ВНЕШНИЕ ЗАВИСИМОСТИ

- **VK Ads API** — доступ запрашивает владелец кабинета (client_id/secret). Агентский статус ИП — открытый вопрос; боевое создание кабинетов — после подтверждения. **Поля API и лимиты креативов меняются несколько раз в год — сверять с живой документацией (context7/офсайт VK), не хардкодить.**
- **kotbot** — API нет; автоматизация интерфейса через Playwright; путь хрупкий → health-check + фолбэк на прямой VK API с обратной связью оператору.
- **Google Forms/Sheets** — чтение брифов через Sheets API; для ПДн клиентов РФ — Яндекс-вариант позже (платно). Слой чтения брифа абстрагирован.
- **Senler** — API; цель «заявка через Senler» через «написать сообщение».
- **n8n** — см. docs/N8N.md; код ходит по REST; воркфлоу едут через git.
- Внешние API — обрабатывать лимиты/ошибки, не падать при недоступности; ретраи где уместно.

## 8. ОПРЕДЕЛЕНИЕ ГОТОВНОСТИ (Definition of Done)

- [ ] Код по конвенциям (раздел 4).
- [ ] Написаны тесты, покрывающие логику шага.
- [ ] `ruff`, `mypy`, `pytest` зелёные локально (фронт: eslint/prettier/tsc).

- [ ] Атомарный коммит с осмысленным сообщением.
- [ ] Запушено в ветку (не в `main` напрямую).
- [ ] CI + Aikido зелёные (при merge в `main`).
- [ ] Отмечено `- [x]` в docs/ROADMAP.md.

Дополнительно для фазы: merge в `main` → автодеплой → `/health` зелёный (раздел 2a).

Если хоть один пункт не выполнен — шаг/фаза не завершены.
